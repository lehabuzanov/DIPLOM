# Описание программной системы электронного корпуса журнала

## Назначение системы

Разработанная система представляет собой веб-приложение для создания, хранения, поиска, анализа и повторного использования электронного корпуса научного журнала «Социально-экономическое управление: теория и практика» ИжГТУ имени М. Т. Калашникова.

Основная задача приложения - предоставить исследователю и редактору единый инструмент для работы с публикациями журнала: от загрузки и структурирования статей до корпусного поиска, формирования выборок, лингвистической обработки текстов и получения аналитических данных.

Система ориентирована на два типа задач:

- библиографические: хранение выпусков, статей, авторов, организаций, ключевых слов, файлов публикаций и ссылок на оригинальный источник;
- корпусно-лингвистические: полнотекстовый поиск, поиск по словоформам и леммам, построение частотных списков, биграмм, сравнение статей и подкорпусов, анализ географии авторов.

Проект реализован как монолитное серверное веб-приложение на Django. Такой подход выбран из-за надежной поддержки реляционных данных, удобной административной панели, простой серверной отрисовки страниц и естественной интеграции с PostgreSQL.

## Текущее состояние корпуса

В текущей версии в систему импортирован архив журнала:

- 21 выпуск;
- 265 опубликованных статей;
- извлеченные тексты для всех 265 статей;
- более 1 млн токенов;
- около 288 авторов;
- географические привязки авторских аффилиаций к городам.

Основной источник наполнения - сайт журнала на платформе OJS:

```text
https://izdat.istu.ru/index.php/social-economic-management
```

Если локальный PDF-файл статьи недоступен, система использует внешний URL статьи или файла на сайте журнала.

## Технологический стек

Backend:

- Python;
- Django 5.2;

- Django ORM;
- Django authentication;
- Django admin.

Развертывание:

## Общая архитектура

Система построена как монолитное Django-приложение с разделением на несколько внутренних приложений:

- `core` - главная страница, страница «О корпусе», инструкция пользователя и общий контекст сайта;
- `accounts` - регистрация, личный кабинет, роли, профили, действия пользователя, сохраненные пометки;
- `corpus` - основная предметная область: журнал, выпуски, статьи, авторы, тексты, токены, леммы, файлы, поиск, подкорпуса, импорт из OJS;
- `analytics` - аналитические панели, частотные списки, биграммы, сравнение материалов, экспорт данных.

Схема взаимодействия:

```text
Пользователь
    -> браузер
    -> Django views
    -> формы и сервисный слой
    -> Django ORM
    -> PostgreSQL
    -> HTML/JSON/CSV-ответ
```

Большая часть страниц отрисовывается на сервере через Django templates. Для интерактивных элементов используются небольшие JavaScript-блоки: подсказки в полях, загрузка длинного текста статьи частями, сохранение пометок, подгрузка аналитических таблиц, работа карты.

## Роли пользователей

В системе предусмотрены три типа пользователей. Роли хранятся в профиле пользователя
и синхронизируются со стандартными группами и разрешениями Django. Благодаря этому
роль влияет не только на отображение пунктов меню, но и на фактический доступ к
административной панели, личным инструментам и редакторским действиям.

Исследователь:

- просматривает каталог выпусков и статей;
- выполняет поиск;
- сохраняет поисковые запросы;
- формирует и редактирует подкорпуса;
- добавляет статьи в подкорпус;
- сохраняет пометки в тексте статьи;
- использует аналитику.

Новый пользователь после регистрации автоматически получает роль исследователя.

Редактор:

- обладает возможностями исследователя;
- получает доступ к ручной загрузке статей;
- может работать с административной частью только в пределах моделей корпуса;
- может пополнять и корректировать корпус;
- не управляет пользователями и ролями.

Администратор:

- обладает полным доступом к Django admin;
- управляет пользователями, ролями и справочниками;
- может корректировать данные корпуса на уровне моделей.

Пароль пользователь может изменить в личном кабинете. Роли редактора и администратора
назначаются администратором через профиль пользователя в Django admin. При подготовке
системы выполняется команда `prepare_access_control`, которая создает роли, группы и
разрешения, но не создает пользователей с публично известными паролями.

Для восстановления доступа предусмотрен сброс пароля через email. В production-режиме
можно включить обязательное подтверждение электронной почты: новый пользователь
создается неактивным, получает ссылку активации и получает доступ к личному кабинету
только после подтверждения адреса.

## Основные вкладки интерфейса

### Главная страница

Главная страница дает краткое представление о корпусе и содержит:

- название электронного корпуса;
- быстрый поиск по корпусу;
- статистику по выпускам, статьям, авторам и токенам;
- последние добавленные статьи;
- краткое описание возможностей корпуса;
- график динамики публикаций по годам.

Главная страница служит входной точкой для пользователя и помогает быстро перейти к поиску, архиву выпусков, аналитике или инструкции.

### О корпусе

Раздел описывает назначение системы, предметную область, состав корпуса и основные сценарии работы. Он выполняет справочную функцию и показывает, что корпус связан с конкретным научным журналом и его публикационным архивом.

### Выпуски

Вкладка «Выпуски» содержит каталог номеров журнала. Для каждого выпуска отображаются:

- год;
- том;
- номер;
- дата публикации, если она доступна;
- количество статей;
- ссылка на детальную страницу выпуска.

На странице выпуска выводится список входящих в него статей. Статьи группируются с учетом разделов, содержат названия, авторов и ссылки на карточки.

### Статьи

Вкладка «Статьи» является библиографическим каталогом публикаций. Она позволяет просматривать все опубликованные статьи корпуса и применять фильтры по метаданным:

- название статьи;
- год;
- том;
- номер выпуска;
- автор;
- язык;
- ключевое слово;
- текстовый фрагмент.

Каталог удобен для навигации по корпусу, когда пользователь ищет конкретную публикацию или хочет быстро оценить состав архива.

### Карточка статьи

Карточка статьи - центральный объект интерфейса. Она содержит:

- название статьи;
- авторскую строку;
- год, том и номер выпуска;
- раздел журнала;
- язык публикации;
- страницы;
- DOI, если он указан;
- дату добавления;
- аннотацию;
- ключевые слова;
- основной очищенный текст;
- ссылки на страницу статьи на сайте журнала и PDF-файл;
- географию авторов, если аффилиации удалось связать с городами;
- похожие статьи из того же раздела;
- форму добавления статьи в подкорпус для авторизованного пользователя.

Текст статьи загружается частями, чтобы длинные публикации не перегружали страницу. Для авторизованных пользователей доступна функция выделения фрагмента текста и сохранения его как личной пометки.

### Поиск

Вкладка «Поиск» предназначена для исследовательской работы с корпусом. Она объединяет текстовый поиск и фильтрацию по метаданным.

Доступные параметры:

- поисковый запрос;
- режим поиска;
- название статьи;
- год от;
- год до;
- том;
- номер;
- автор;
- язык;
- ключевое слово.

Режимы поиска:

- полнотекстовый поиск;
- поиск по лемме;
- поиск по словоформе;
- поиск по фразе.

Если пользователь вводит текстовый запрос, система ищет совпадения в корпусе и показывает контексты. Если текстовый запрос не указан, поиск работает как фильтр по метаданным и возвращает все статьи, соответствующие условиям.

Результат поиска содержит:

- количество найденных статей;
- название статьи;
- год и раздел;
- авторов;
- релевантность для полнотекстового поиска;
- число совпадений;
- фрагменты контекста с подсветкой.

Результаты можно экспортировать в CSV. Авторизованный пользователь может сохранить запрос или сформировать из результатов подкорпус.

### Сохраненные запросы

Сохраненный запрос - это набор параметров поиска, который пользователь может повторно открыть. Для каждого запроса сохраняются:

- название;
- описание;
- параметры фильтрации;
- последнее количество результатов.

Пользователь может:

- открыть сохраненный запрос;
- изменить название и описание;
- удалить запрос;
- вернуться к нему из личного кабинета.

### Подкорпуса

Подкорпус - это сохраненная исследовательская выборка статей. Он нужен для повторного анализа выбранной группы материалов.

Подкорпус может формироваться двумя способами:

- автоматически по фильтрам из поиска;
- вручную путем добавления отдельных статей из карточки статьи или со страницы подкорпуса.

Вкладка «Подкорпуса» показывает список сохраненных пользователем выборок. Для каждого подкорпуса отображаются:

- название;
- описание;
- количество статей;
- количество токенов;
- сохраненные фильтры;
- признак публичности;
- кнопки открытия, аналитики, изменения фильтров, параметров и удаления.

### Страница подкорпуса

Страница «Открыть» показывает основное состояние подкорпуса без перегрузки формами:

- название;
- описание;
- статистику по статьям и токенам;
- дату последнего обновления;
- источник формирования;
- текущие сохраненные условия отбора;
- блок ручного добавления статьи;
- состав подкорпуса;
- источник каждой статьи: добавлена по фильтру или вручную;
- кнопку удаления статьи только из данного подкорпуса;
- возможность вернуть последнюю удаленную статью.

Удаление статьи из подкорпуса не удаляет статью из корпуса и не затрагивает общий архив. Удаляется только связь между статьей и конкретным подкорпусом.

### Изменение фильтров подкорпуса

Отдельная страница «Изменить фильтры» содержит расширенное управление составом подкорпуса. Она включает:

- текущую статистику;
- сохраненные условия отбора;
- форму изменения фильтров;
- кнопку применения фильтров;
- кнопку обновления состава по сохраненным фильтрам;
- ручное добавление статьи;
- список статей подкорпуса.

При применении новых фильтров система пересобирает часть состава, полученную по фильтру. Статьи, добавленные вручную, сохраняются. Это позволяет сочетать автоматическую выборку с индивидуальной корректировкой исследователя.

### Параметры подкорпуса

Страница «Параметры» отвечает за служебные свойства подкорпуса:

- название;
- описание;
- публичность.

Фильтры изменяются не здесь, а на отдельной странице изменения фильтров, чтобы разделить описание подкорпуса и работу с его составом.

### Пометки

Пометки доступны авторизованному пользователю. На странице статьи пользователь может выделить фрагмент текста и сохранить его.

В разделе «Пометки» отображаются:

- статьи, в которых есть сохраненные фрагменты;
- выделенные фрагменты;
- комментарии к пометкам;
- возможность редактировать комментарий;
- возможность удалить пометку;
- переход к нужному месту в тексте статьи.

Функция предназначена для исследовательской разметки и последующего возвращения к важным цитатам.

### Карта

Вкладка «Карта» показывает географию авторов по аффилиациям. Система связывает организации авторов с городами с помощью встроенного справочника и нормализации названий.

На странице карты доступны:

- фильтр по году;
- фильтр по городу;
- интерактивная карта;
- маркеры городов;
- различение российских и зарубежных городов;
- статистика по городам, авторам, организациям и статьям;
- лидеры по публикационной активности;
- распределение по странам;
- список статей для выбранного города;
- динамика активности по годам.

Географический модуль полезен для анализа научных связей журнала и распределения авторов по регионам.

### Аналитика

Вкладка «Аналитика» предназначена для количественного анализа корпуса, отдельной статьи или подкорпуса.

Доступные режимы:

- анализ всего корпуса;
- анализ выбранной статьи;
- анализ выбранного подкорпуса;
- сравнение двух статей;
- сравнение двух подкорпусов.

Показатели:

- динамика публикаций по годам;
- самые активные авторы;
- частотный список лемм;
- частотный список словоформ;
- биграммы;
- общие единицы двух сравниваемых материалов;
- частоты по материалу A и материалу B;
- число совпадений.

Для частотных списков предусмотрены два режима фильтрации:

- очищенный режим, скрывающий технический шум, DOI, EDN, короткие формы, цифры, библиографические элементы;
- полный режим, показывающий весь поток токенов без дополнительной фильтрации.

Данные аналитики можно экспортировать в CSV.

### Загрузка статьи

Раздел «Загрузка статьи» доступен редактору или администратору. Он позволяет вручную добавить публикацию в корпус.

Форма включает:

- год выпуска;
- том;
- номер;
- заголовок выпуска;
- дату публикации;
- раздел журнала;
- название статьи;
- язык;
- авторов и аффилиации;
- ключевые слова;
- аннотацию;
- основной текст;
- список литературы;
- страницы;
- DOI;
- URL оригинала;
- файл статьи;
- признак опубликованности.

При сохранении система создает или обновляет выпуск, раздел, авторов, аффилиации, статью, файл, текст статьи и ключевые слова.

### Личный кабинет

Личный кабинет объединяет пользовательские данные и историю работы:

- сохраненные запросы;
- сохраненные подкорпуса;
- количество пометок;
- недавняя активность;
- переходы к основным пользовательским разделам.

Для редактора дополнительно доступен переход в административный раздел.

### Административная панель

Django admin используется для внутреннего управления данными. Через нее можно просматривать и редактировать:

- журналы;
- выпуски;
- разделы;
- статьи;
- авторов;
- аффилиации;
- ключевые слова;
- файлы;
- тексты;
- токены;
- леммы;
- подкорпуса;
- запросы;
- пользователей;
- роли.

Административная панель особенно полезна для редактора корпуса и технического сопровождения.

## Структура базы данных

База данных построена вокруг нескольких групп моделей.

### Библиографический слой

`Journal` - журнал, к которому относится корпус. Хранит полное и краткое название, ISSN, издателя, описание, сайт журнала и признак активности.

`Issue` - выпуск журнала. Содержит год, том, номер, заголовок выпуска, дату публикации, ссылку на источник и служебное примечание.

`Section` - раздел журнала. Используется для тематической группировки статей внутри выпуска.

`Article` - основная модель публикации. Хранит журнал, выпуск, раздел, название, slug, язык, страницы, DOI, оригинальный URL, аннотацию, источник импорта, внешний идентификатор, признак опубликованности.

`ArticleAuthor` - промежуточная модель между статьей и автором. Фиксирует порядок автора, отображаемое имя и аффилиацию в рамках конкретной статьи.

`Author` - автор публикации. Хранит имя, фамилию, отчество, slug, ORCID, email и связанные организации.

`Affiliation` - организация автора. Хранит название, город, страну, сайт, географическую привязку и уровень уверенности геокодирования.

`Keyword` - ключевое слово. Хранит исходное и нормализованное значение, а также язык.

`ArticleFile` - файл статьи. Хранит локальный файл, тип файла, исходное имя, MIME-тип, внешний URL и checksum.

### Текстовый и лингвистический слой

`ArticleText` - структурированный текст статьи. Содержит текст заголовка, аннотации, ключевых слов, основной текст, список литературы, очищенный полный текст, количество токенов, количество лемм и поисковый вектор PostgreSQL.

`Lemma` - лемма токена. Хранит нормализованную форму, язык и часть речи.

`ArticleToken` - токен статьи. Содержит исходную форму, нормализованную форму, ссылку на лемму, позицию, номер предложения, границы символов, источник в тексте и морфологическую информацию.

Этот слой обеспечивает поиск по словоформам, леммам, построение частотных списков и контекстов.

### Географический слой

`CityLocation` - город или населенный пункт, используемый для географической привязки аффилиаций. Содержит отображаемое и нормализованное название, регион, страну, координаты, источник геокодирования и признаки проверки.

Города связываются с аффилиациями авторов и используются на странице карты.

### Пользовательский слой

`Role` - роль пользователя с правами на управление пользователями, редактирование контента, запуск импортов и сохранение запросов.

`UserProfile` - профиль пользователя. Хранит организацию, должность, предпочитаемый язык и основную роль.

`UserActivity` - журнал пользовательских действий: поиск, работа с подкорпусом, экспорт, импорт, вход.

`SearchHistory` - история поисковых запросов с типом поиска, фильтрами и числом результатов.

`SavedQuery` - сохраненный запрос пользователя. Содержит название, описание, JSON-параметры запроса и последнее число результатов.

`SavedSubcorpus` - сохраненный подкорпус. Хранит пользователя, название, описание, JSON-фильтры, публичность, число статей, число токенов и связь со статьями.

`SavedSubcorpusArticle` - связь статьи и подкорпуса. Дополнительно хранит источник добавления: по фильтру или вручную.

`ArticleHighlight` - личная пометка пользователя в статье. Хранит диапазон символов, выделенный текст и комментарий.

## Импорт архива OJS

Система поддерживает автоматическую синхронизацию с архивом журнала на OJS.

Команда:

```powershell
docker compose exec web python manage.py sync_ojs_journal --skip-existing
```

Основные этапы:

1. Загрузка страницы архива выпусков.
2. Извлечение ссылок на выпуски.
3. Разбор страницы каждого выпуска.
4. Извлечение ссылок на статьи.
5. Разбор страницы статьи и meta-тегов.
6. Получение названия, авторов, аффилиаций, DOI, страниц, аннотации, ключевых слов и списка литературы.
7. Загрузка PDF-файла.
8. Извлечение текста из PDF.
9. Очистка текста.
10. Создание или обновление записей в БД.
11. Построение токенов, лемм и поискового индекса.

Если PDF не удается загрузить или разобрать, система пытается сохранить хотя бы метаданные статьи. Для восстановления текстов предусмотрена отдельная команда:

```powershell
docker compose exec web python manage.py repair_article_texts
```

## Очистка и индексирование текста

После извлечения из PDF текст может содержать технические элементы: номера страниц, заголовки, DOI, сведения об авторах, библиографические блоки, переносы слов. Для подготовки текста используются функции очистки:

- нормализация пробелов;
- удаление мягких переносов;
- склейка переносов внутри слов;
- отсечение служебных заголовков;
- поиск начала содержательного текста;
- исключение библиографических хвостов;
- удаление дублирующихся фрагментов заголовка и аннотации.

После очистки текст разбивается на предложения и токены. Для каждого токена сохраняются:

- исходная форма;
- нормализованная форма;
- лемма;
- часть речи;
- морфологический тег;
- позиция;
- границы в тексте;
- раздел текста.

Для русских слов используется `pymorphy3`, для английских - `simplemma`. Исторические кириллические символы дополнительно нормализуются.

## Поисковая подсистема

Поиск работает в нескольких режимах.

Полнотекстовый поиск:

- использует PostgreSQL full-text search;
- применяет поисковый вектор `ArticleText.search_vector`;
- возвращает релевантность;
- формирует фрагменты контекста с подсветкой.

Поиск по фразе:

- использует phrase search PostgreSQL;
- подходит для поиска устойчивых выражений.

Поиск по словоформе:

- ищет конкретную нормализованную форму токена;
- возвращает контексты вокруг найденных позиций.

Поиск по лемме:

- нормализует пользовательский запрос;
- ищет все токены, связанные с нужной леммой;
- позволяет находить разные грамматические формы одного слова.

Поиск по метаданным:

- применяется, когда текстовый запрос не указан;
- фильтрует статьи по году, выпуску, автору, языку, ключевому слову и названию;
- используется для формирования подкорпусов.

## Подкорпуса как исследовательский механизм

Подкорпус позволяет сохранить выбранный набор статей и возвращаться к нему в дальнейшем. Это важно для исследовательских задач, где нужно анализировать не весь корпус, а тематическую, временную или авторскую выборку.

Пример:

```text
Год от: 2020
Год до: 2022
Язык: Русский
```

Такой набор фильтров сохраняется в `SavedSubcorpus.filter_payload`. При обновлении состава система заново применяет фильтры и пересобирает связи со статьями, добавленными по фильтру. Ручные добавления при этом сохраняются.

Подкорпус хранит:

- исходные фильтры;
- список статей;
- источник включения каждой статьи;
- счетчик статей;
- счетчик токенов.

Счетчики пересчитываются напрямую по связям в БД, чтобы избежать рассинхронизации между сохраненными числами и фактическим составом.

## Аналитическая подсистема

Аналитика строится на токенах и леммах, сохраненных в БД.

Частотные списки:

- могут строиться по словоформам или леммам;
- исключают список литературы;
- в очищенном режиме удаляют шумовые единицы;
- поддерживают поиск по таблице;
- подгружаются частями через JSON.

Биграммы:

- строятся по последовательностям значимых токенов;
- используются для выявления устойчивых сочетаний.

Сравнение:

- сравнивает две статьи или два подкорпуса;
- показывает общие единицы;
- выводит частоты в материале A и материале B;
- сортирует по числу совпадений, частоте и алфавиту.

Экспорт:

- частотные списки, биграммы и сравнение можно выгружать в CSV.

## География авторов

Географический модуль основан на аффилиациях авторов. Система пытается определить город организации по встроенному справочнику и нормализованным подсказкам.

Для обновления географии используется команда:

```powershell
docker compose exec web python manage.py refresh_author_geography
```

Дополнительно может использоваться внешний геокодер:

```powershell
docker compose exec web python manage.py refresh_author_geography --geocode
```

Результаты внешнего геокодирования помечаются как требующие проверки.

## Файловое хранение

Файлы статей хранятся через Django `FileField` в media-хранилище. В Docker-развертывании media вынесена в отдельный volume.

Для каждой статьи может храниться:

- локальный PDF;
- исходное имя файла;
- тип файла;
- внешний URL;
- MIME-тип.

Если локальный файл отсутствует, пользователь перенаправляется на внешний URL статьи или PDF.

## Интеграция с сайтом журнала

Проект может быть внедрен на сайт журнала как отдельный раздел или как самостоятельный поддомен. Рекомендуемый вариант:

- развернуть Django-приложение на сервере;
- подключить PostgreSQL;
- настроить media-хранилище;
- настроить сбор статических файлов;
- использовать production WSGI/ASGI-сервер;
- настроить reverse proxy;
- запускать импорт OJS по расписанию;
- периодически пересобирать географию и индекс при необходимости.

Текущий импорт зависит от структуры сайта OJS. Пока сохраняются архивная страница, ссылки на выпуски, ссылки на статьи и meta-теги, синхронизация работает устойчиво. При существенном изменении HTML-разметки сайта журнала парсер потребуется адаптировать.

## Практическая ценность

Система объединяет функции электронного архива, поискового инструмента и корпусно-аналитической платформы.

Для редакции журнала она полезна как:

- цифровой архив публикаций;
- инструмент навигации по выпускам;
- средство повторного использования текстов;
- база для дальнейшей интеграции с сайтом журнала.

Для исследователей она полезна как:

- источник структурированных научных текстов;
- инструмент поиска по метаданным и содержанию;
- средство построения подкорпусов;
- инструмент лемматизированного поиска;
- средство частотного анализа;
- инструмент сравнения материалов;
- средство фиксации личных пометок.

Для лингвистических и междисциплинарных задач система дает возможность анализировать лексику, устойчивые сочетания, динамику публикаций, авторскую географию и тематические выборки научного журнала.
